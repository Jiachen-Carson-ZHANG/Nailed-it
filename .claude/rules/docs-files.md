# Docs editing rules — formatting reference

Fires when editing anything under `docs/`. Companion to `doc-discipline.md` (procedure rule). This file covers **what to write**; doc-discipline.md covers **when to write it**.

## General

- Keep docs concise and concrete.
- Prefer current system truth over historical narration.
- Image references use standard markdown `![](../screenshots/name.png)` from `docs/changes/`. Standard markdown renders in both Obsidian and GitHub.
- Cross-doc links use standard markdown `[label](FILE.md)` — not Obsidian wiki-links.

## PATCHES.md — per-entry template

Each patch entry uses this exact structure. Insert at the TOP of the matching area section (newest-first within area).

```markdown
### AREA-X# — Short title
**Type:** Enhancement | Feature · **Status:** FIXED YYYY-MM-DD · **Priority:** P#

- **What:** one-sentence problem or capability statement
- **Why it matters:** business / user impact
- **Fix applied:**
  - (backend): change 1
  - (frontend): change 2
- **Trade-off:** when applicable
- **Must remain true:** invariants future devs must preserve

**Before**
![](../screenshots/YYYY-MM-DD-area-id-before.png)
*italic caption describing the before state*

**After**
![](../screenshots/YYYY-MM-DD-area-id-after.png)
*italic caption describing the after state*

---
```

Omit the `**Before**` / `**After**` blocks for non-visual changes (Bucket B — no placeholder images, no broken refs).

## PATCHES.md — anchor format (GitHub-style)

Anchors are auto-generated by GitHub from heading text. Construct manually with this rule:

1. Take the heading text after `### ` (e.g. `UI-E16 — Proceed button tweak`).
2. Lowercase everything.
3. Drop punctuation except spaces and hyphens.
4. Replace whitespace runs with a single `-`.
5. The em-dash strips to nothing → leaves a double-hyphen between ID and title.
6. Prefix with `#`.

Example: `### UI-E16 — Proceed button tweak` → `#ui-e16--proceed-button-tweak`.

## PATCHES.md — Index row format

Index is sorted by Priority (P0 → P3), then newest-first within each priority. Insert the new row at the TOP of the matching priority bucket.

```
| **P#** | 🔧 Enhancement | [AREA-X#](#anchor) | Title | YYYY-MM-DD |
```

Type icon: `🔧 Enhancement` for E, `✨ Feature` for F.

## CHANGELOG.md — bullet format

One bullet per shipped change, under today's `## YYYY-MM-DD` section. Newest date section at the top.

```
- 🔧 Enhancement · <Area Full Name> · [AREA-X#](PATCHES.md#anchor) — <Title>
```

`<Area Full Name>` matches the area section heading from PATCHES.md.

## BACKLOG.md — row format

One row per open item, sorted by Priority.

```
| **P#** | **AREA-X#** | Area Full Name | OPEN / PENDING | Title |
```

When the item ships, remove the row entirely. Do not strike-through or mark as DONE — that lives in CHANGELOG + PATCHES.

## Frozen ID rule

IDs in `PATCHES.md` and `BACKLOG.md` are frozen — `AREA-E#` / `AREA-F#` once assigned, never renumbered. Type can be corrected in the entry body (E ↔ F) but the ID letter stays. Gaps in numbering are intentional history.

Use `python3 .claude/skills/record-patch/scripts/scan_ids.py` to get the next-available ID. The `next` value is the only valid choice for that area+kind.

## ADRs in docs/decisions/

Follow the template in `docs/decisions/README.md`: Context · Decision · Design principles · Alternatives considered · Consequences (positive / negative) · References.

File naming: `ADR-####-short-title-in-kebab-case.md`. Zero-padded sequential. ADR numbers are frozen — never renumber. If a decision is reversed, write a new ADR that supersedes the old one rather than editing history.

## Architecture snapshot

`docs/architecture/current-state.md` is the current system shape. When the design changes materially:

1. Move the current "Pipeline shape" / "Subsystems" content into a `## Previous (YYYY-MM-DD)` section at the bottom.
2. Write the new state at the top.

This preserves the diff trail for future audits.
